# 🔴 Critical Findings

## CRY-001 — Unauthenticated Encryption (no AEAD/MAC)

| Attribute | Value |
|---|---|
| Severity | Critical |
| CWE | CWE-353 (Missing Support for Integrity Check); CWE-345 (Insufficient Verification of Data Authenticity); CWE-924 (Improper Enforcement of Message Integrity) |
| Locations | `lib/src/cipher.rs:32-43,118-129`; `lib/src/entry/write.rs:113-165`; `lib/src/entry/read.rs:14-62`; `lib/src/chunk/types.rs:104-116` (no PMAC/PHMC chunk exists) |

### Description
Both CBC and CTR mode pipelines are pure confidentiality primitives with no MAC/AEAD wrapper. `lib/Cargo.toml` has no `hmac`, `aead`, `aes-gcm`, `chacha20poly1305`, `poly1305`, `cmac`, or `gmac` dependencies (verified). The chunk type list contains only `PHSF` for crypto metadata — no chunk carrying a tag, HMAC, or MAC over `(IV || FDAT*)`. CRC32 in `RawChunk` (`chunk.rs`) only protects bit-flip transit errors, not adversarial modification.

### Reproduction
1. **CTR bitflip**: any byte in FDAT after an attacker-known plaintext range XORs the plaintext at that exact offset with attacker-chosen bits — trivial silent tampering; libpna surfaces modified bytes with no error.
2. **Padding-oracle on CBC** (Vaudenay-style): Send victim a doctored archive; observe whether decryption returns `InvalidData` (Pkcs7 unpad failure at `lib/src/cipher/block/read.rs:101`) versus succeeds and downstream observes garbled bytes. The error path is distinguishable.
3. **Algorithm-substitution**: rewrite Encryption byte in FHED from CTR(=1) to CBC(=0); reader at `entry/read.rs:38-58` honors the chunk-declared mode, decrypts with wrong mode → leak via error/no-error.

### Impact
- **Confidentiality**: chosen-ciphertext / padding-oracle plaintext recovery on CBC
- **Integrity**: silent CTR bitflip / arbitrary-block CBC corruption
- **Availability**: forced parse failures
- Applies to **every encrypted libpna archive ever produced**.

### Recommendation
- **Long-term**: Migrate primary path to a true AEAD (AES-256-GCM, AES-256-GCM-SIV for nonce-misuse resistance, or XChaCha20-Poly1305).
- **Short-term** (preserve format compat): Encrypt-then-MAC construction:
  - Derive separate MAC key via HKDF from Argon2 output
  - Compute HMAC-SHA-256 over `(algorithm || mode || PHSF || IV || all FDAT chunks)`
  - Store in new critical chunk (e.g. `PMAC`)
  - Reject entry if verification fails *before* exposing any plaintext
  - Use `subtle::ConstantTimeEq` for the comparison
  - Bump format major version (old readers must refuse new archives and vice-versa)

### References
- Vaudenay 2002 "Security Flaws Induced by CBC Padding" (EUROCRYPT)
- Bellare-Namprempre 2000 "Authenticated Encryption: Relations among Notions"
- CVE-2013-0169 (Lucky13)
- CVE-2016-2107 (OpenSSL AES-NI CBC padding oracle)
- RFC 5116 (AEAD interface)
- RustCrypto `aead` crate

# 🟠 High Findings

## CRY-002 — CBC Padding Oracle (timing/error distinguishable)

| Attribute | Value |
|---|---|
| Severity | High |
| CWE | CWE-208 / CWE-203 (Observable Discrepancy) |
| Locations | `lib/src/cipher/block/read.rs:100-101`; `lib/src/cipher.rs:131-141` |

### Description
On the final block, `Pkcs7::unpad` returns `Err` iff PKCS#7 padding is invalid. That error becomes `io::ErrorKind::InvalidData` and is surfaced to the caller. Successful unpad followed by garbled plaintext takes a *different* code path. An attacker who can submit chosen ciphertexts and observe `InvalidData` vs success can mount Vaudenay's attack and recover plaintext one byte at a time.

### Reproduction
Attacker wraps target ciphertext as final block, prepends crafted IV-like block, asks victim CLI to extract; observes process exit code or stderr. ~128 oracle queries per byte recovered.

### Impact
Full plaintext recovery against CBC archives where the attacker can repeatedly invoke the victim's decrypt path on chosen ciphertexts. Realistic in:
- "Decrypt every uploaded archive" services
- IDE preview plugins
- Shared-archive sync tools

### Recommendation
Same root fix as CRY-001 (AEAD or Encrypt-then-MAC). If neither is feasible immediately, **deprecate CBC mode**: mark `CipherMode::CBC` in `lib/src/entry/options.rs:323` as `#[deprecated]`, default to CTR, refuse to write new CBC archives.

> ⚠️ Switching to CTR fixes the padding-oracle but does **NOT** fix integrity (CRY-001 still applies — CTR bitflips remain trivial).

### References
- Vaudenay EUROCRYPT 2002
- Lucky13 CVE-2013-0169
- POODLE CVE-2014-3566

## CRY-003 — Algorithm/Mode Downgrade Attack

| Attribute | Value |
|---|---|
| Severity | High |
| CWE | CWE-757 (Selection of Less-Secure Algorithm); CWE-345 |
| Locations | `lib/src/entry/read.rs:14-62`; `lib/src/entry/options.rs:294-339`; `lib/src/hash.rs:54-84` |

### Description
The decrypt path reads `encryption` and `cipher_mode` from the entry header and `algorithm` from the PHSF string, then uses the password to derive the key. Nothing binds those choices to the password or the ciphertext. An attacker who controls archive bytes can flip:
- `Encryption::Aes` ↔ `Encryption::Camellia`
- `CipherMode::CTR` ↔ `CipherMode::CBC`
- PHSF prefix from `$argon2id$` to `$pbkdf2-sha256$` with weakened iteration count

### Reproduction
Build CTR archive, edit single FHED byte to change mode to CBC, ship to victim → victim's reader attempts CBC decrypt giving attacker-controlled padding oracle (CRY-002).

### Impact
- Confidentiality / integrity (enables CRY-002 even on CTR-original archives)
- Long-term key-strength downgrade for offline crackers

### Recommendation
Bind `(encryption, cipher_mode, PHSF, IV)` into the MAC from CRY-001's fix. Derive encryption key via HKDF with the algorithm-mode-version string mixed into `info`, so any tampering produces a wrong key (not just a wrong mode).

### References
- CVE-2015-3197 (FREAK/Logjam family)
- RFC 7905

## PARSE-001 ≡ MUL-001 — `archive_number + 1` Overflow (Cross-validated)

| Attribute | Value |
|---|---|
| Severity | High |
| CWE | CWE-190 (Integer Overflow); CWE-345; CWE-754 |
| Locations | `lib/src/archive/read.rs:144`; `lib/src/archive/read/slice.rs:153`; `lib/src/archive/write.rs:246` |
| Validation | **Independently discovered by 2 agents** — confidence very high |

### Description
`archive_number` is `u32` (`header.rs:12`). `read_next_archive` validates multipart linkage with `current_header.archive_number + 1 != next.header.archive_number`. The workspace `Cargo.toml` has no `overflow-checks = true`; release builds inherit standard overflow semantics.

In **release builds**: addition silently wraps. When attacker supplies a "current" archive whose AHED archive_number is `u32::MAX = 0xFFFFFFFF`, `current + 1` becomes `0`, so the check passes for a "next" archive whose archive_number is `0`.

Mirror flaw on writer side (`write.rs:246`): `let next_archive_number = self.header.archive_number + 1;` — wraps to 0 silently.

In **debug builds**: same operation panics → DoS.

### Reproduction
```
Part 1: PNA header + AHED with archive_number = 0xFFFFFFFF
        (8-byte AHED body: 00 00 00 00 FF FF FF FF) + ANXT
Part 2: PNA header + AHED with archive_number = 0
        (8-byte AHED body: 00 00 00 00 00 00 00 00) + AEND
Caller: archive1.read_next_archive(part2_reader)
```
Release-built consumer: `+1` wraps to 0, equality holds, unrelated part2 accepted.
Debug build: panic on integer overflow.

### Impact
- **Authenticity bypass** for multipart chaining — attacker substitutes forged "next" archive
- **Debug-build DoS** via panic on integer overflow

### Recommendation
- Use `current_header.archive_number.checked_add(1).ok_or_else(|| io::Error::new(InvalidData, "archive number overflow"))?`
- Apply same fix at all 3 sites (writer + 2 readers)
- Reject any AHED `archive_number = u32::MAX` at parse time (no legitimate archive can have a successor)
- **Estimated fix effort: 5 minutes**

### References
- CWE-190
- Analogous to CVE-2018-8740 class issues in archive parsers

## MUL-002 — No Archive-Set Identity Binding (Substitution Attack)

| Attribute | Value |
|---|---|
| Severity | High |
| CWE | CWE-345 (Insufficient Verification of Authenticity); CWE-353 (Missing Integrity Check) |
| Locations | `lib/src/archive/header.rs:9-13`; `lib/src/archive/read.rs:140-154`; `lib/src/archive/write.rs:245-254` |

### Description
AHED carries only `(major, minor, archive_number)` plus 2 ignored "general purpose bit flag" bytes (`header.rs:43` explicitly drops them). There is no archive-set UUID, no random nonce, and no archive-wide digest. `read_next_archive` only checks `current.archive_number + 1 == next.archive_number`. CRC32 covers each chunk individually but nothing binds part N to part N+1 of the same logical archive.

### Reproduction
Attacker has access to part2 of two unrelated multipart PNA archives A and B (both produced with `pna create --split`). Both have AHED with archive_number=2. Substitute B.part2 for A.part2 in transit/storage. Reader links them, walks chunk stream across boundary; if entry boundaries align, entries from B are silently delivered as if from A.

### Impact
Silent content substitution in multipart sets. Particularly dangerous when parts traverse separate channels (cloud storage of part1, local part2). Backups/snapshots tampered without detection.

### Recommendation
1. Allocate the 2 currently-reserved AHED bytes (or extend AHED) to carry an **archive-set identifier** (random 64/128-bit nonce) generated at `Archive::write_header` and propagated through `split_to_next_archive`
2. Reader must reject mismatched IDs in `read_next_archive`
3. **Specification update needed** in `chunk_specifications/index.md §4.1.1`
4. Long-term: add optional MAC chunk binding the full chunk-CRC sequence per part, anchored by archive-set ID

### References
- ZIP multi-disk uses identical disk-set signature
- CWE-345

## UNS-010 — Public API Soundness Hole (set_len + read_exact)

| Attribute | Value |
|---|---|
| Severity | High |
| CWE | CWE-908 (Use of Uninitialized Resource); CWE-704 (Incorrect Type Conversion) |
| Locations | `lib/src/cipher/block/read.rs:35` |

### Description
In `CbcBlockCipherDecryptReader::new`:
```rust
let mut buf = ArrayVec::new();
debug_assert_eq!(block_size, buf.capacity());
unsafe { buf.set_len(buf.capacity()) };
r.read_exact(&mut buf)?;
```

`set_len(capacity)` makes all 16 slots logically initialized while their backing `MaybeUninit<u8>` are NOT actually initialized. The next line passes `&mut buf` to `Read::read_exact`.

**The `Read` trait contract permits an implementation to read FROM the supplied buffer before writing**. Since Rust 1.60+ the API contract has been clarified: passing uninitialized memory to `Read::read` is **undefined behavior**. Any `Read` implementation that calls `buf.iter()`, `buf[i] += x`, or copies via SIMD with read-modify-write on bytes beyond what it overwrites can observe the uninit value → UB.

### Reproduction
Custom `Read` impl whose `read(&mut self, buf)` does `let _ = buf[0];` before writing. With this reader, instantiating any `aes-cbc` / `camellia-cbc` decryption reads uninit memory → UB. Under MIRI this would trip.

### Impact
Soundness violation in libpna's **public API surface**. A safe-Rust caller (custom Read impl) can trigger UB without writing `unsafe` themselves — this is a **soundness bug in libpna**.

### Recommendation
Two safe options:
1. Initialize: `for _ in 0..block_size { buf.push(0); }` (one-time, 16-byte cost)
2. Use `Read::read_exact` against an already-zeroed `[u8; 16]` then `try_extend_from_slice`

Either eliminates the unsafe entirely.

**Add MIRI test**: `cargo +nightly miri test -p libpna cipher`.

**Estimated fix effort: 5 minutes** + ~30 minutes for MIRI CI integration.

### References
- Rustonomicon §10
- rust-lang/rust #42788 (Read+uninit)
- clippy::uninit_vec
- RUSTSEC-2018-0003 pattern

## RES-001 — Decompression Bomb (xz/zstd/zlib unlimited)

| Attribute | Value |
|---|---|
| Severity | High |
| CWE | CWE-409 (improper handling of compressed input); CWE-1284 (improper input validation) |
| Locations | `lib/src/entry/read.rs:64-80`; `lib/src/compress.rs:86-107` |

### Description
`decompress_reader()` wraps a `BufReader` in flate2/zstd/liblzma decoders without:
- Any `io::Read::take()` limit
- Any output-counting wrapper
- Any upper bound derived from the `fSIZ` chunk

The `DecompressReader` enum and its `Read` impl forward `read()` unchanged. Consumers calling `io::copy` / `read_to_end` will read until decoder reports EOF. All three backends accept maximum legal stream sizes (zstd 2^64-1 frame content, deflate ~unbounded, xz multi-stream concat). No `fSIZ` cross-check is performed against actual decoded bytes.

### Reproduction
- **zstd**: 32-byte frame describing 2^60 zeros → ~2^55 amplification
- **xz**: ~1 KiB (LZMA2 RLE of zeros) → 100 GiB output (amplification 10^8 on null payloads)
- **zlib**: 1 MiB of `0x00` deflated → ~1 KiB; reverse for ~1000× amplification

Malicious archive needs only one FDAT chunk, no encryption, no other chunks. Victim runs `pna x foo.pna` or any program calling `entry.reader(...).read_to_end(&mut Vec::new())` → exhausts RAM / fills disk.

### Impact
OOM kill of process / disk-fill DoS / wall-time DoS for any tool consuming attacker-controlled archives (CI, mail attachment scanners, cloud workers). Affects every libpna consumer including `pna` CLI.

### Recommendation
- (a) Cap decompressed output via `Read::take()` driven by configurable `ReadOption "max_decompressed_size"` (default e.g. 16 GiB), and/or
- (b) When `fSIZ` is present, wrap decoder in a counting reader that errors after `fSIZ` bytes (currently `fSIZ` is parsed into `Metadata.raw_file_size` but never enforced)
- (c) Provide explicit `DecompressReader::with_limit` constructor

### Dependency CVE Status (verified)
- `liblzma 0.4.6` / `liblzma-sys 0.4.3` bundles **xz-utils 5.8.1** — clean of CVE-2024-3094 (backdoor was in 5.6.0/5.6.1 release tarball build script only, not src/liblzma)
- `zstd 0.13.3` / `zstd-sys 2.0.14+zstd.1.5.7` bundles **zstd 1.5.7** — post CVE-2022-4899 fix
- `flate2 1.1.9` defaults to `miniz_oxide 0.8.5` (pure-Rust, no native CVE)

### References
- CWE-409
- xz bomb folklore; zip bomb 42.zip
- zstd Decoder docs note `ZSTD_d_windowLogMax` can be limited via `ZSTD_DCtx_setParameter` — not used here

## CRY-004 — No Memory Zeroization

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-226 (Sensitive Information in Resource Not Removed); CWE-244 |
| Locations | `lib/src/entry/options.rs:276-290` (`Password(Vec<u8>)` with Clone, no Drop); `lib/src/entry/write.rs:19-29` (`CipherContext`); cipher state in `lib/src/cipher/block/{read,write}.rs` |

### Description
Passwords (`Vec<u8>` in `Password`/`ReadOptions`/`WriteOptions`), the Argon2/PBKDF2 32-byte derived key (`password_hash::Output`), and the IV all sit in heap allocations freed without overwriting. Heap dump, swap file, or use-after-free in another part of the process can recover them.

`Cargo.toml` lacks `zeroize` (verified by grep). RustCrypto cipher types do not zero their key schedule on drop unless wrapped with `Zeroizing<>`.

### Reproduction
`gcore` on a `pna` extract process post-decryption → `strings` → recover password and derived key.

### Recommendation
- Add `zeroize` (and ideally `subtle`) to `lib/Cargo.toml`
- Wrap `Password`, `Output`, IV, and intermediate copies in `Zeroizing<Vec<u8>>` or implement explicit `Drop`
- Document CLI-level recommendation to read passwords from stdin/file rather than argv

### References
- CWE-226
- "How to safely zero memory in Rust" (Diem 2019)
- `zeroize` crate docs
- OWASP ASVS V6.2.4

## CRY-005 — Argon2id Default Below RFC 9106 First-Recommended

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-916 (Use of Password Hash With Insufficient Computational Effort) |
| Locations | `lib/src/hash.rs:6-32`; `lib/src/entry/write.rs:84-105` |

### Description
When user does not specify Argon2 parameters (default in `WriteOptionsBuilder` via `HashAlgorithm::argon2id()` at `lib/src/entry/options.rs:425-428`), crate falls back to `argon2 0.5.3` defaults: `m_cost=19456 KiB (≈19 MiB)`, `t_cost=2`, `p_cost=1`.

RFC 9106 §4 defines two recommended profiles:
- **First** (memory-constrained): 64 MiB / t=3 / p=4
- **Second** (low-memory devices): 19 MiB / t=2 / p=1

The library defaults to the *low-memory* profile, which RFC 9106 explicitly says should only be used "if much less memory is available." For desktop/server backups, the higher profile is appropriate.

### Impact
Confidentiality of weak passwords against offline GPU/ASIC attackers — increases practical brute-force speed by ~3-4× compared to RFC first-recommended profile.

### Recommendation
Override default in `HashAlgorithm::argon2id()` to pin `m_cost=65536 (64 MiB), t_cost=3, p_cost=4` (or `available_parallelism()`). Keep argon2 0.5.3 defaults available via `_with(None,None,None)` opt-out only when consumer explicitly requests low-memory mode.

### References
- RFC 9106 §4 "Recommendations"
- OWASP Password Storage Cheat Sheet (2023)

## CRY-006 — PBKDF2-SHA256 Weak, No Downgrade Protection

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-916 |
| Locations | `lib/src/hash.rs:34-43`; `lib/src/entry/options.rs:382-407`; `lib/src/entry/write.rs:98-104` |

### Description
PBKDF2 default (600,000 rounds via pbkdf2 0.12.2 `Params::default()`) matches OWASP 2023 minimum, so iteration count itself is OK. However, PBKDF2 has **no memory-hardness** and is ~10,000× faster on GPUs than Argon2id.

Doc comments at `options.rs:367-385` mark Argon2id as preferred, but downgrade to PBKDF2 still derives a *full-strength key* and writes a normal PHSF — there is no archive-level signal warning the recipient that a weaker KDF was chosen.

Combined with **CRY-003** (KDF downgrade), this lets an attacker rewrite a future archive's PHSF prefix to PBKDF2-SHA256 / minimum rounds, then crack offline far faster.

### Recommendation
Either:
- Deprecate PBKDF2 entirely for new archives (only allow read), or
- Refuse to write PBKDF2 unless user passes opt-in flag

At minimum, raise PBKDF2 minimum rounds enforced by the library (separate from pbkdf2 crate's default) to match Argon2id's effective cost.

## CRY-007 — CTR Keystream RNG Failure Detection

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-323 (Reusing a Nonce); CWE-338 (Use of Cryptographically Weak PRNG) — partial |
| Locations | `lib/src/random.rs:8-17`; `lib/src/entry/write.rs:38,46-48` |

### Description
`random::random_bytes` instantiates a fresh `ChaCha20Rng::from_entropy()` for every call. `from_entropy` pulls from `getrandom` (OS CSPRNG) — fundamentally fine. However:
- On platforms where `getrandom` can fail or return low-entropy seed early in boot (Linux pre-5.6 `getrandom(0)` blocking semantics, embedded targets, WASM `js` feature in Cargo.toml that delegates to `crypto.getRandomValues`), the per-call construction pattern hides any catastrophic seeding failure
- Each call's RNG is independent, so a brief seed-collision window cannot be detected by re-checking the same RNG

CTR-mode key-stream reuse → XOR of two plaintexts trivially recovers both.

### Recommendation
- Use a single process-global `OsRng` (or thread-local) with `try_fill_bytes` so any RNG failure is surfaced as an error rather than silently degrading
- Defense-in-depth: include a counter or per-entry random nonce in IV-derivation HKDF together with `salt`, so even RNG repeat cannot produce same IV unless `(salt, counter)` also collide

### References
- Joux "Authentication failures in NIST version of GCM" 2006
- Linux random.c BUG history (`getrandom` pre-5.6)
- CVE-2008-0166 (Debian OpenSSL)

## PARSE-002 ≡ RES-003 — `set_max_chunk_size` Bypass (Cross-validated)

| Attribute | Value |
|---|---|
| Severity | Medium (High when combined with RES-001) |
| CWE | CWE-789 (Memory Allocation with Excessive Size); CWE-770; CWE-1284 |
| Locations | `lib/src/chunk/read.rs:160` (`read_chunk_from_slice`); `lib/src/archive/read/slice.rs:34,57`; `lib/src/archive/read.rs:50,203` (AHED uses `ChunkReader::new(reader, None)` — header chunk NEVER bounded); `lib/src/archive.rs:72,87,107` (default `max_chunk_size = None`) |
| Validation | **Independently discovered by 2 agents** |

### Description
`Archive::set_max_chunk_size` documents protection: "Chunks larger than this size will be rejected with an error, protecting against maliciously crafted archives with extremely large chunks." This is implemented in the `Read` path via `allocate_chunk_data(length, max_chunk_size)`.

However:
1. The slice-based path (`read_chunk_from_slice`) takes **no `max_chunk_size` parameter at all**
2. AHED is read with `ChunkReader::new(&mut reader, None)` — the very first user-controlled length field bypasses the limit
3. CLI never calls `set_max_chunk_size` (verified `grep -rn set_max_chunk_size cli/src/` returns no matches)
4. Default `max_chunk_size = None` → effectively `MAX_CHUNK_DATA_LENGTH = u32::MAX (≈4 GiB)`

### Reproduction
PNA header (`89 50 4E 41 0D 0A 1A 0A`) followed by AHED with length `FF FF FF FF` (4 GiB) + type `41 48 45 44`. `try_reserve_exact(4_294_967_295)` either succeeds (server with abundant RAM) or returns `Err`. Even if allocation succeeds, a single 4 GiB request per active connection is trivially DoS.

### Recommendation
1. Plumb `max_chunk_size` through `read_chunk_from_slice` and `read_header_from_slice`
2. Apply same `max_chunk_size` to AHED read in both `read_header_with_buffer` and `read_header_from_slice_with_buffer`
3. Document a **default ceiling** (e.g., 64 MiB)
4. Bound per-entry chunk count in `next_raw_item` / `next_raw_item_slice`

### References
- CWE-789; libarchive CVE-2024-26458 class

## PARSE-003 — 32-bit `compressed_size` Overflow

| Attribute | Value |
|---|---|
| Severity | Medium (Low on 64-bit) |
| CWE | CWE-190 |
| Locations | `lib/src/entry.rs:636,660`; `lib/src/entry/meta.rs:24` |

### Description
`compressed_size` is a `usize`, accumulated as `compressed_size += chunk.data().len();`. With many `FDAT` chunks each carrying near `u32::MAX` bytes, on a 32-bit consumer (wasm32, armv7) this addition wraps silently in release mode. The wrapped value is exposed via `Metadata::compressed_size()` and consumed by callers that may use it to size buffers.

### Recommendation
- Widen `compressed_size` to `u64`, OR
- Use `saturating_add` and document saturation, OR
- Reject input whose cumulative compressed bytes exceed `usize::MAX` / plumbed limit

## PARSE-004 ≡ RES-005 — Chunks-per-Entry Unbounded (Cross-validated)

| Attribute | Value |
|---|---|
| Severity | Medium (Low–Medium) |
| CWE | CWE-770 |
| Locations | `lib/src/archive/read.rs:69-87`; `lib/src/archive/read/slice.rs:52-73`; `lib/src/entry.rs:247-271` |
| Validation | **Independently discovered by 2 agents** |

### Description
Inside one entry (between FHED and FEND) there is no upper bound on the number of chunks. Code accumulates them into `Vec<RawChunk>`. Each empty chunk is 12 bytes on disk + ~32 bytes resident.

An archive of ~600 MiB containing a single FHED → 50 million empty `xATR`/extra chunks → FEND uses ~1.6 GiB of RSS during parse. Slice path is worse: one Vec push per chunk amplifies a 12-byte input into ~40 bytes resident.

### Reproduction
PNA + AHED + FHED + (12-byte empty `xUNK` chunk repeated N times) + FEND + AEND. Attacker controls N up to ~file_size/12.

### Recommendation
- Add `max_chunks_per_entry` configuration (default ~64K)
- Short-circuit in `next_raw_item`/`next_raw_item_slice`/`read_next_normal_entry_from_stream` when exceeded
- Consider global archive-level chunk limit

### References
- CWE-770; libarchive billion-laughs analogue

## PARSE-005 — Chunk Ordering Invariants Not Enforced

| Attribute | Value |
|---|---|
| Severity | Medium (Low) |
| CWE | CWE-707 (Improper Neutralization); CWE-345 |
| Locations | `lib/src/archive/read.rs:48-60, 69-88`; `lib/src/archive/read/slice.rs:32-72` |

### Description
Specific gaps:
- Second `AHED` chunk inside an entry is silently swallowed → "Unknown critical chunk type" error (defensive but type-confusion possible)
- `ANXT` chunk between FHED and FEND merely sets `next_archive = true` silently (`read.rs:79`, `slice.rs:64`) — the entry continues to read past it, malicious archive can mark "next archive present" mid-entry to confuse caller's `has_next_archive()` state
- `SHED` before `AEND` followed by `AEND` without `SEND` → `try_from` for `SolidEntry` iterates remaining chunks looking for `SEND` and treats `AEND` as "unknown critical" — misleading error message masks underlying truncation

### Reproduction
- (a) **ANXT confusion**: PNA + AHED + FHED + ANXT + FEND + AEND. Reader returns one entry and `has_next_archive() == true`; caller proceeds and may consume unrelated bytes
- (b) **Solid truncation**: PNA + AHED + SHED + AEND. Iterator returns `Err("Unknown critical chunk type: AEND")` instead of truncation diagnostic

### Recommendation
- In entry loop, treat `ANXT` between FHED and FEND as error
- In `SolidEntry::try_from`/`NormalEntry::try_from`, distinguish "unexpected archive terminator" from "unknown critical chunk"

### References
- CWE-707, ambiguous parsing class (CVE-2007-1860 family)

## RES-002 — Unbounded ANXT Multipart Chain Depth

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-674 (uncontrolled recursion / iteration); CWE-770 |
| Locations | `lib/src/archive/read.rs:140-154`; `lib/src/archive/read/slice.rs:152` |

### Description
`read_next_archive` validates only that `next.archive_number == current+1`. `archive_number` is u32, so a coordinated set of crafted part files could exercise up to 2^32 sequential reads. Each part allocates fresh `ChunkReader` buffers, propagates `self.buf (Vec<RawChunk>)` of unbounded raw chunks to next part, and never releases carried buffer cap.

### Recommendation
- Add configurable max-parts limit (default 1024)
- When carrying `self.buf` across `read_next_archive`, drop entries that have been emitted (they appear to be unconsumed-tail-of-prior-part chunks and should be bounded)
- Enforce sane upper bound on chain length even when unset

## RES-004 — SolidEntry Materializes All SDAT in RAM

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-770 |
| Locations | `lib/src/entry.rs:307-312` (SolidEntry struct); `lib/src/entry.rs:524-583` (try_from collects all SDAT into `Vec<T>`); `lib/src/entry.rs:432-446` (entries() / into_entries()) |

### Description
**The CLAUDE.md design note "Archives never need to be fully resident in memory" is violated for solid mode.** `NormalEntry::data` and `SolidEntry::data` are `Vec<T>` of every FDAT/SDAT chunk for the entry. `RawEntry::try_from → {Normal,Solid}Entry::try_from` collects them all before returning. There is no streaming SolidEntry path.

With `max_chunk_size` at u32::MAX (PARSE-002/RES-003) and no limit on SDAT count, an archive with N SDAT chunks of size S forces N×S resident bytes per single-entry parse.

### Reproduction
Archive with one SHED + 256 × 16 MiB SDAT + one SEND forces ~4 GiB resident before `SolidIntoEntries` can yield even the first inner entry.

### Recommendation
- True streaming SolidEntry path that decompresses SDAT chunks on the fly without retaining them, by combining `ChunkReader` + `decrypt_reader` + `decompress_reader` on the underlying reader directly (rather than materializing chunks first), OR
- Cap aggregate (S)DAT bytes per entry via config knob

## UNS-009 — ArrayVec set_len in Encrypt Path (Latent Issue)

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-908 |
| Locations | `lib/src/cipher/block/write.rs:50` |

### Description
In `encrypt_write_with_padding`, `self.buf` may be partially filled (0..block_size). `unsafe { self.buf.set_len(block_size) }` extends logical length up to `block_size` (= 16). Bytes between previous `pos` and `block_size` are uninitialized `u8`. Immediately after, `P::pad(block, pos)` is called.

Soundness invariants required:
1. `ArrayVec::set_len(N)` requires N <= capacity AND elements 0..N initialized when later read as `T`. Capacity is fixed at 16, AES-256 / Camellia-256 block_size is 16. **In release builds without `debug_assert`, any cipher with block_size != 16 would set_len > capacity → instant UB**
2. `Padding::pad` from RustCrypto writes `block[pos..block_size]`. So uninitialised tail is fully written before any read
3. Today no observable UB; latent — any change to cipher type or padding scheme can flip to memory corruption / RCE-class UB

### Recommendation
Replace with safe equivalent:
```rust
let pos = self.buf.len();
while self.buf.len() < block_size { self.buf.push(0); }
```
Or use `ArrayVec::extend_from_slice(&[0u8; 16][..block_size - pos])`. Both avoid unsafe with negligible perf cost. If retaining `unsafe`, add `// SAFETY:` comment AND turn debug_assert into hard `assert!`.

### References
- Rustonomicon §10
- clippy::uninit_vec

## MUL-003 — Truncated/Missing Final Part Not Detected

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-755 (Improper Handling of Exceptional Conditions); CWE-252 |
| Locations | `lib/src/archive/read.rs:69-88`; `cli/src/command/core.rs:1316-1340` |

### Description
Reader treats AEND as "end of archive" and returns `Ok(None)`. Library API `has_next_archive` only returns whether ANXT was seen *after iterator exhaustion*. Callers using libpna directly without re-checking after the loop may consume part1, see no next-archive flag, and silently believe archive is complete.

If attacker strips ANXT but keeps AEND in a non-final part, libpna reports complete; nothing flags missing data.

### Reproduction
Multipart archive of 3 parts. Attacker deletes ANXT (4-byte length=0 + "ANXT" + 4-byte CRC = 12 bytes) from part1 trailer. AEND remains. Reader treats part1 as complete; entries in part2/part3 lost.

### Impact
Silent partial extraction. Particularly dangerous for backups.

### Recommendation
- Make ANXT presence mandatory metadata bound by integrity (per MUL-002 fix), OR
- Emit warning whenever `archive_number > 0` but consumed file lacks ANXT and is not preceded by ANXT-bearing predecessors
- Document threat in `recommendations_for_decoders/index.md`

## MUL-004 — AEND Not Enforced (Truncation Oracle)

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-130 (Improper Handling of Length Parameter Inconsistency) |
| Locations | `lib/src/archive/read.rs:69-88`; `lib/src/chunk/read.rs:120-158` |

### Description
Read loop terminates *only* on AEND (returns `Ok(None)`) or I/O error. Entries already produced before truncation point are returned as success and silently extracted by typical consumers using `for entry in archive.entries()` that only break on `Err`. Library callers that swallow errors get partial extraction.

### Reproduction
Truncate any complete PNA at chunk boundary, before AEND. Run `pna list` — outputs partial entry list followed by error; library callers that swallow errors get partial extraction.

### Recommendation
Provide `Archive::is_finalized()` after iteration, OR have iterator yield sentinel `Ok(None)` only if AEND was actually seen. Add test that truncated-at-boundary archive surfaces explicit error.

## MUL-005 — AHED Inconsistency Across Parts Not Validated

| Attribute | Value |
|---|---|
| Severity | Medium |
| CWE | CWE-757 (Selection of Less-Secure Algorithm During Negotiation) |
| Locations | `lib/src/archive/read.rs:140-154` (read_next_archive) |

### Description
`read_next_archive` only checks next part's `archive_number`. Next part's AHED `(major, minor, general_purpose_bits)` is read into new `ArchiveHeader` and silently replaces current header (line 142-143). Currently spec defines only major/minor/archive_number, but as format evolves, this gap allows silent feature-flag downgrades or upgrades inside single logical archive.

### Reproduction
Construct part2 with major=1 (hypothetical future). Reader accepts and applies part1's parser semantics to part2's chunks.

### Impact
Future-proofing risk; in combination with MUL-002, attacker can flip GP-flag bits (currently `bytes[2..4]` are dropped in `from_bytes`) to encode side-channel state.

### Recommendation
In `read_next_archive`, assert `current_header.major == next.header.major && current_header.minor == next.header.minor`. When GP-bit flags become defined, also assert equality. Persist and validate currently-ignored bytes 2..4 of AHED.

## MUL-006 — CRC32 Not Authenticated (PNG Convention but Doc Gap)

| Attribute | Value |
|---|---|
| Severity | Low |
| CWE | CWE-345 |
| Locations | `lib/src/chunk/read.rs:124-150, 161-199`; `lib/src/chunk/crc.rs` |

### Description
CRC32 covers `(type + data)` but not length — fits PNG convention. CRC32 itself is ~2^16 collision-resistant; for adversarial integrity it is inadequate. Consistent with PNG spec but should be documented as non-goal for security-grade integrity. Currently `lib/src/chunk/crc.rs` simply re-exports `crc32fast::Hasher` with no comment about its security properties.

### Recommendation
- Add doc comment in `crc.rs` explicitly stating CRC32 is for transport-error detection, not authentication
- Encryption modes already provide confidentiality but lack integrity (CTR has no MAC, CBC is malleable). Consider HMAC chunk for AEAD-equivalent semantics
- Document that integrity for encrypted archives requires external signing

### References
- PNG spec §5.3 (CRC excludes length, same convention)

## MUL-007 — Unconsumed-Chunk Buffer Carries Across Part Boundary

| Attribute | Value |
|---|---|
| Severity | Low |
| CWE | CWE-665 (Improper Initialization) |
| Locations | `lib/src/archive/read.rs:69-88` (chunks pushed to `self.buf` on AEND), 142 (`read_header_with_buffer(reader, self.buf)` carries buf across boundary) |

### Description
On AEND in part N, accumulated chunks (between FHED..FEND/SEND) are pushed back to `self.buf`. `read_next_archive` passes that buffer to part N+1's reader. Intent: entry split across ANXT boundary completes in part N+1 — but code only handles case where entry is *complete* in part N (FEND/SEND seen). If entry is in-progress (no FEND seen) and AEND appears (malformed but not rejected — see PARSE-005), unconsumed chunks transfer to part N+1 and may be misinterpreted as part of unrelated entry.

### Recommendation
In `next_raw_item`, on AEND, assert `chunks.is_empty()` (or only buffered chunks are recognized continuation markers). Reject otherwise.

## PAN-001 — Cipher block_size Invariant ArrayVec (Defense in Depth)

| Attribute | Value |
|---|---|
| Severity | Info |
| CWE | CWE-617 (Reachable Assertion) — false positive |
| Locations | `lib/src/cipher/block/read.rs:78, 111`; `lib/src/cipher/block/write.rs:52,72,80,84,95,102` |

### Description
`<&Block<C>>::try_from(self.buf.as_slice()).expect("buf length equals block size")`. `self.buf` is `ArrayVec<u8, 16>` set via `set_len(capacity)` in `new()`. Only consumed via `cipher.rs:35,43` aliases (Aes256/Camellia256, both block_size=16). `debug_assert_eq!` on construction would catch a future block_size mismatch.

Not reachable today — algebraic invariant: ArrayVec capacity (16) == AES/Camellia block size (16). Becomes Critical if a 32-byte-block cipher is added without resizing ArrayVec.

### Recommendation
Use const generic `ArrayVec<u8, BS>` parameterized by `cbc::Decryptor::<C>::block_size()`, or switch to runtime `Box<[u8]>` of `block_size()` length to make invariant structural rather than hidden coupling.
